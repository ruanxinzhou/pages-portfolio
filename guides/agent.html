🤖 Agent 设计指南

设计高质量 AI Agent 的标准原则与实践

解决什么问题？

🚫 问题 1：上下文腐烂（Context Rot）

当对话越来越长，AI 会逐渐"忘记"早期的上下文，开始产生不一致的回答。就像你和一个人聊了 3 小时，他开始混淆之前说过的话。

表现：AI 忘记了你之前定义的变量名、重复问已经回答过的问题、生成的代码与之前的约定矛盾。

🚫 问题 2：上下文窗口限制

AI 的上下文窗口是有限的（比如 200K tokens）。当项目代码量超过这个限制，AI 无法同时"看到"所有代码，导致理解不完整。

表现：AI 只能处理局部代码，无法理解全局架构；修改一个文件时破坏了另一个文件的逻辑。

🚫 问题 3：需要人在回路中

简单任务也需要人一步步指挥："先执行这个，再执行那个"。效率低下，且容易遗漏步骤。

🚫 问题 4：任务复杂度爆炸

大型任务（如"重构整个认证系统"）涉及多个文件、多个步骤，手动管理容易出错。

✅ Agent 的解决方案

Agent 通过自主规划、步骤拆解、工具调用、自我修正，将复杂任务自动化。它能管理自己的上下文，按步骤执行，并在出错时自动修复。

Agent 核心概念

Agent 是一个能自主规划并执行任务的 AI 系统。它不仅仅是"问答机器"，而是能够：

  • 理解目标：解析用户的高层意图
  • 拆解步骤：将复杂任务分解为可执行的子任务
  • 调用工具：使用外部工具完成具体操作
  • 检查修正：自我验证并修复错误
角色定位设计

每个 Agent 都应该有清晰的角色定义，就像团队中的岗位职责：

# Agent 角色定义模板 ## 角色名称 你是一个 [具体角色]，负责 [核心职责]。 ## 核心能力 - 能力 1：[具体描述] - 能力 2：[具体描述] - 能力 3：[具体描述] ## 工作方式 当你收到任务时，你会： 1. [步骤 1] 2. [步骤 2] 3. [步骤 3]

示例：代码审查 Agent

code-reviewer.md
## 角色名称 高级代码审查专家 ## 核心职责 对提交的代码进行全面审查，确保代码质量、安全性和可维护性。 ## 审查维度 1. 正确性：代码逻辑是否正确 2. 安全性：是否存在安全漏洞 3. 性能：是否有性能问题 4. 可读性：代码是否易于理解 5. 规范性：是否遵循团队规范
边界与约束

明确的边界定义可以防止 Agent 越权或做出不当行为：

绝对禁止（ABSOLUTELY FORBIDDEN）

🚫 严禁事项

这些是 Agent 绝对不能做的事情：

绝对禁止
# 绝对禁止 - **严禁**直接修改 `src/models/` 目录下的文件 - **严禁**在未获得用户确认的情况下删除任何文件 - **严禁**执行可能导致数据丢失的命令（如 `rm -rf`） - **严禁**将敏感信息（密码、密钥）输出到日志 - **严禁**绕过质量门控（Gate Check）直接进入下一阶段

必须遵守（MUST）

✅ 强制要求

这些是 Agent 必须做到的事情：

必须遵守
# 必须遵守 - **必须**在每次操作前备份相关文件 - **必须**使用中文与用户沟通 - **必须**在完成任务后生成操作日志 - **必须**遇到不确定时主动询问用户 - **必须**遵循现有的代码规范和架构模式
输入输出规范

清晰的 I/O 规范可以让 Agent 更准确地理解任务和产出结果：

输入规范

输入格式规范
# 输入格式 每个任务必须包含： 1. 任务 ID：唯一标识符（如 TASK-2024-001） 2. 任务描述：清晰的目标说明 3. 上下文：相关的背景信息 4. 约束条件：必须遵守的限制 # 输入示例 { "task_id": "TASK-2024-001", "description": "重构用户认证模块，支持 OAuth2.0", "context": "当前使用 JWT，需要兼容旧版本", "constraints": ["不能修改数据库表结构", "必须向后兼容"] }

输出规范

输出格式规范
# 输出格式 { "task_id": "TASK-2024-001", "status": "completed | failed | blocked", "result": { "summary": "任务完成摘要", "changes": ["修改的文件列表"], "artifacts": ["产出的文档或代码"] }, "next_steps": ["建议的后续操作"], "blockers": ["遇到的阻塞问题（如有）"] }
错误处理策略

Agent 需要具备完善的错误处理能力：

错误处理流程
# 错误处理流程 1. 捕获错误：识别错误类型和严重程度 2. 分类处理： - 可自愈错误 → 自动重试（最多 3 次） - 可恢复错误 → 执行修复流程 - 不可恢复错误 → 上报用户决策 3. 记录日志：完整记录错误信息和处理过程 4. 通知用户：重要错误必须通知用户

重试策略

重试配置
# 重试配置 max_retries: 3 retry_delay: 2s exponential_backoff: true # 重试条件 - 网络超时 → 重试 - API 限流 → 等待后重试 - 语法错误 → 自动修复后重试 - 逻辑错误 → 上报用户
质量门控（Gate Check）

每个阶段完成后必须通过质量检查才能进入下一阶段：

Gate Check 配置
# Gate Check 配置 Gate 0: 知识搜集完成 - ✓ 后端代码已拉取 - ✓ 前端图谱 ≥ 3 节点 - ✓ CODE_KNOWLEDGE.md 已生成 Gate 1: 需求澄清完成 - ✓ 唯一任务 ID - ✓ 需求可验证 - ✓ 用户已确认 Gate 2: 方案设计完成 - ✓ TECHNICAL_SPEC.md 已生成 - ✓ feature/*.md 已生成 - ✓ Headless 架构合规
